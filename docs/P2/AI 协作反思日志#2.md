## 2. AI 助力分析（Q1）
本阶段 AI 最大的贡献是：快速生成多视角的架构方案，激发团队批判性讨论。

不同 Prompt 策略下 AI 回答质量的差异：在架构辩论赛中，不同Prompt策略下AI回答质量的差异非常明显。成员A直接问“推荐什么架构”，得到的往往是教科书式的通用方案，忽略了团队新手、10周工期等现实约束；成员B提供了详细的项目约束后，AI能准确识别关键限制，给出可直接落地的单体分层架构建议；成员C要求对比多种方案，AI会输出结构化的多维权衡表，便于讨论但若脱离具体背景仍偏理论；成员D让AI扮演架构师，回答逻辑完整却可能过早关注扩展性。综合来看，只有将具体约束与对比需求注入Prompt，才能使AI从泛泛而谈转向量身定制，这是提升架构建议实用性的核心结论。

## 3. AI 误导分析（Q2）
AI 在架构建议中的典型问题：
- 问题1：AI 容易给出“看起来完整但偏理想化”的架构建议，倾向于过早强调微服务、消息队列、复杂中间件和高扩展性设计，而没有充分考虑我们当前只是 4 人团队、10 周周期、代码基础还比较初始的现实约束。如何发现：在对照课程文档要求和现有仓库结构后，我们发现 AI 给出的部分建议虽然技术上合理，但明显超出了当前项目阶段的实现成本。例如仓库目前就是前后端分离的基础项目结构，后端仍以简单 Spring Boot 应用为主，如果直接引入微服务拆分、网关、消息队列和复杂缓存，会显著增加开发、部署、调试和协作成本，与“10 周内可行性”这一核心约束不匹配。如何修正：人工最终将架构收敛为“前后端分离 + 单体分层架构”的方案，即前端采用 Vue，后端采用 Spring Boot，内部再按控制层、服务层、数据访问层分层，并以 RESTful API 进行交互。对于 MySQL、缓存、消息队列等能力，我们只保留为后续扩展选项，而不在当前 P2 阶段作为核心架构前提。这样既保留了系统后续演进空间，也保证了当前阶段的可实现性和可维护性。


## 4. Prompt 改进计划（Q3）
上阶段（P1）提出的改进措施，验证结果：上阶段提出“在Prompt中明确项目约束、团队能力及任务范围”的改进措施，经本阶段实践验证，AI输出与项目实际的适配度明显提高，过度设计的无效建议减少，人工修改幅度平均下降，协作效率有所提升，但在AI输出与本人核心工作职责的贴合度上仍需优化，部分内容需人工调整才能匹配进程内轻量异步相关工作。

本阶段新的改进措施：每次提交Prompt前，增加“明确输出重点和简洁格式”的要求，避免AI输出冗余内容，进一步降低人工修改和梳理的工作量。

## 5. 本阶段的核心工程判断
决策内容：应该选用什么架构风格

为什么 AI 无法做这个决策：AI无法代替我们站在团队的角度从项目背景、团队能力、任务范围、项目约束等信息中判断出最合适的架构风格，只能给出各个风格的架构方案的优缺点。只有我们在经过讨论之后才能选出真正适合我们进行开发并且能够满足整个项目需要的架构方案和开发技术。
